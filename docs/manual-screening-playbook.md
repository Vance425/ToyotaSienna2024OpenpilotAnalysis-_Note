# 人工篩選操作手冊

## 目標

把目前的研究從「知道哪些 ID 可疑」推進到「每次拿到 log 都知道怎麼看」。

## 使用方式

每次分析都分成兩條線一起看：

- 候選線：`0x260 / 0x116 / 0x115 / 0x131`
- 參考線：`0x610 / 0xD5`

不要只看單一 ID，也不要只看單一場景。

## 先看哪些場景

優先順序：

1. `comma 3X` 失敗前後
2. `IGN off -> IGN on`
3. `IGN on -> Ready`
4. 原廠 `ACC on/off`
5. 原廠 `ADAS on/off`

## 逐 ID 觀察指南

### 0x260

角色：

- 目前最核心的被動候選
- 很像 control / state / protected frame family

優先看：

- bytes `1,2`
- bytes `2,3`
- bytes `5,6`
- bytes `6,7`

要問的問題：

- 在 `comma 3X` 失敗前後，哪一組 bytes 最先改變？
- 在狀態切換時，它是平滑改變、跳變，還是 reset？
- 高熵 tail 是否在某些窗口突然失去連續性？

### 0x116

角色：

- 強 SecOC-like 候選
- 重點是 tail 很像 auth / counter 區

優先看：

- bytes `4,5,6,7`

要問的問題：

- 同一個 core 下，tail 是否每次都變？
- tail 在 `IGN on` 或 `Ready` 時是否 reset？
- 失敗窗口內是否出現停頓、重複、明顯模式切換？

### 0x115

角色：

- 最早期就很強的 counter / auth-like 候選
- 很適合和 `0x116 / 0x131` 一起對照

優先看：

- bytes `4,5,6,7`

要問的問題：

- 它和 `0x116 / 0x131` 是同步變化，還是互補變化？
- 某些 bits 是否在固定場景特別活躍？
- 是否存在明顯的 rolling / auth pattern？

### 0x131

角色：

- 穩定前排候選
- 很像 control/state 與 auth-like 的交界訊號

優先看：

- bytes `1,2`
- bytes `2,3`
- bytes `4,5`
- bytes `5,6`

要問的問題：

- 左右轉、ADAS 狀態切換時，哪一段在動？
- 高熵區塊是跟 steering/state 走，還是獨立變動？
- 與 `0x115 / 0x116` 是否同時出現異常窗口？

## 參考線怎麼用

### 0x610

角色：

- steering reference

用途：

- 判斷當下是不是在轉向
- 對齊候選 frame 的變化時刻

### 0xD5

角色：

- speed reference

用途：

- 判斷當下是否有車速變化
- 避免把正常速度引起的變化誤判成 security 事件

## 哪些變化最值得標記

- reset
- 突然停頓
- 明顯跳變
- 某段 bytes 從高熵變低熵
- 某段 bytes 在失敗前後切換模式
- 多個候選 ID 同時出現變化

## 最實際的分析順序

1. 先用 `0x610 / 0xD5` 對齊場景
2. 再看 `0x260`
3. 再對照 `0x116 / 0x115 / 0x131`
4. 最後記錄哪個時間窗口最可疑

## 每次分析的最小產出

至少留下三件事：

1. 一個關鍵時間窗口
2. 一組最可疑的 bytes
3. 一句下一輪可驗證的假設
