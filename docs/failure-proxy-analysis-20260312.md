## 目的

現有資料沒有明確的 `comma 3X` 錯誤時間戳，所以這一輪用 session 邊界當 proxy，檢查模型是否在段落切換前後出現「斷裂點」。

代理窗口：

- $label (local-only source path) 尾端
- [toyota_seg_IGN_ON_20260312_190101_000.ndjson](../logs/toyota_seg_IGN_ON_20260312_190101_000.ndjson) 開頭

參考事件：

- `185801` stop: `1773342054714`
- `190101` start: `1773342060646`
- `190101` ignition_transition: `1773342061644`

## 觀察 1: `185801` 尾端

尾端 `~350 ms` 內，模型沒有亂掉，但明顯收斂：

- `0x116` phase 已經固定到 `00 00`
- `0x131` 前段維持在 `0015 -> 0018 -> 0019 -> ... -> 002a`
- `0x260` 前段集中在 `0000050080...`，只有小幅擾動
- `0xD5` 維持 `08000000000000e5`

### 判讀

這不像 protected frame 失去同步，更像：

- session 正在收斂到某個基底狀態
- `0x116` phase 已經退到最低層

## 觀察 2: `190101` 開頭

開頭 `~350 ms` 內，一樣從低層狀態重新起跑：

- `0x116` phase 仍從 `00 00` 開始
- `0x131` 前段從 `001d0046 -> 001e0046 -> 00210047 -> ... -> 00320053`
- `0x260` 從 `00ffd90046...` 漸進到 `0000050053...`
- `0x610` 在 `1773342061778` 出現 anchor
- `0xD5` 保持 `08000000000000e5`

### 判讀

這不是模型重新建立失敗，而是：

- session restart 後
- family state 從基底狀態往上展開
- `0x116` 暫時維持 `00 00`，等後面再進入更高 phase

## 最重要的結論

目前在這個 failure proxy 視角下，沒有看到「模型突然斷裂」。

看到的是：

1. session 尾端時，`0x116` phase 收斂到 `00 00`
2. 新 session 開頭時，`0x116` 也從 `00 00` 重新起跑
3. `0x131 / 0x260` 仍維持有序的 family state 演進

## 這代表什麼

如果 `comma 3X` 的真正問題是拿不到可用安全材料，那目前更像：

- 整個安全相關 family 在 session 邊界時會回到基底狀態
- 但被動 log 裡沒有看到任何能把它推進到「可用解鎖狀態」的額外事件

換句話說：

- 不是 frame 結構壞掉
- 是目前看到的都是「正常 protected family 的基底/運作態」
- 但沒有看到真正讓 `comma 3X` 成功跨過安全門檻的訊號

## 對後續分析的意義

這一輪把方向再縮小了一步：

- 不要再找「哪個 frame 壞掉」
- 要找的是「什麼事件能讓 `00 00` 基底 phase 之外的 session 狀態被打開」
