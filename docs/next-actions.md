# 下一步執行清單

## 目標

透過被動 CAN log 分析，把 `TSK` 問題縮小成明確可驗證的候選 frame 與時序。

## 先收集的資料

- [ ] 門框貼紙製造年月
- [ ] VIN 首碼
- [ ] 市場版本
- [ ] trim 等級
- [ ] 目前使用的線束型號
- [ ] 目前使用的軟體 / fork 名稱
- [ ] 目前看到的錯誤訊息或症狀
- [ ] 至少一組原始 CAN log
- [ ] 第二輪被動採集計畫已建立

## 現場檢查

- [ ] 確認裝置是否穩定上電
- [ ] 確認線束是否接對且插緊
- [ ] 確認原廠 ACC / LTA 本身正常
- [ ] 確認沒有任何 ADAS 警示燈
- [ ] 記錄裝置畫面上的提示文字
- [ ] 記錄 CAN log 採集時的車輛狀態
- [ ] 依照 `round-2-capture-plan.md` 執行第二輪被動採集
- [ ] 優先觀察 `0x116 / 0x177 / 0x260`

## 研究分類

### 類型 A: 缺 baseline log

先補原廠純車況 log，否則很難做差異分析。

### 類型 B: 有 log，但看不出關鍵訊息

優先做時序分析、狀態切換分析、候選 frame 的 tail bytes 分析。

### 類型 C: 已鎖定可疑訊息，但還不能證明和 TSK 有關

優先累積重複測試與不同條件下的對照資料。

## 推進順序

1. 先收齊 baseline 與失敗 log
2. 再做差異與時序分析
3. 再縮小疑似 `TSK / SecOC` 候選 frame
4. 最後才進入方法驗證
