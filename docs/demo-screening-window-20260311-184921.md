# 第二輪示範: 事件窗口分析

## 分析對象

- Log: [toyota_seg_IGN_ON_20260311_184921_000.ndjson](/D:/Codex/toyota-sienna-tsk-analysis/analysis-input/raw_can_logs/toyota_seg_IGN_ON_20260311_184921_000.ndjson)
- bus: `0`
- 事件窗口: `1773255027000` 到 `1773255027400`

## 為什麼選這個窗口

因為在這個窗口內：

- `0x610` 發生了明顯變化
- `0xD5` 幾乎完全不變

這很適合示範如何用參考線判斷：

- 現在比較像 steering / state 事件
- 不是 speed 事件

## 先看參考線

### 0x610

在窗口前後：

- `1773255026572`: `0000003cd0fffd40`
- `1773255027173`: `0080043cd009e540`

判讀：

- 這不是微小抖動，是一個明顯狀態改變
- 可以把 `1773255027173` 視為這輪分析的錨點

### 0xD5

在整個窗口裡維持：

- `08000000000000e5`

判讀：

- 這個窗口裡沒有明顯速度狀態改變
- 所以候選線的變化，比較不像是車速本身造成

## 再看候選線

### 0x116

判讀：

- 前四 bytes 有階段性遞進：`26 -> 27 -> 28 -> 29`
- 後四 bytes 持續高變化
- 很像「前半有狀態 / counter 階段，後半是 auth-like 動態區」

### 0x115

窗口前後樣本：

- `1773255027141`: `075400026b793c09`

- 後半段仍然明顯高熵
- 和 `0x116` 同步變，但結構沒有 `0x116` 那麼乾淨

### 0x131

### 0x260

窗口前後樣本：

- `1773255027141`: `000047003d0101f0`
- `1773255027163`: `000049003d00f6e6`
- `1773255027183`: `00004c003d013226`

- `1773255027383`: `00005b0042027881`

判讀：

- 多個 byte 群同時平滑往前走
- 很像 control/state family 的核心 frame
- 不像單純尾巴在變，而是前中後都有結構性移動

## 這個窗口最重要的觀察

### 1. 這是 steering / state 事件，不像 speed 事件

原因：

- `0x610` 明顯改變
- `0xD5` 完全不變

### 2. 候選線不是亂變，而是一起有結構地往前推進

- `0x116` 有階段性遞進
- `0x131` 有狀態式遞進
- `0x260` 有多欄位同步平滑變化
- `0x115` 與它們同步，但更像混合型高熵 frame

### 3. 0x116 在這個窗口裡仍然最像乾淨的 protected 候選

因為：

- 前半段呈現可讀的階段遞進
- 後半段保持高熵
- 結構最像「前半 state/counter，後半 auth-like」

## 第二輪示範結論

這個窗口讓我們學到的不是 `TSK` 已經找到，而是：

- 我們已經有能力用參考線切出事件窗口
- 也有能力看出候選線在窗口內是同步變化，還是單獨亂變

就這個窗口來說，目前最值得記住的是：

1. `0x610` 可以當事件錨點
2. `0xD5` 可以排除 speed 因素
3. `0x116` 是這個窗口裡最乾淨的 protected 候選
4. `0x260` 更像核心 control/state family
5. `0x131` 和 `0x115` 像跟著一起變的輔助候選

## 下一輪假設

如果之後找到 `comma 3X` 失敗時間更明確的窗口，應該優先驗證：

- `0x116` 的前半段是否出現異常階段跳變
- `0x115 / 0x131` 是否和 `0x116` 同步 reset 或脫節
- `0x260` 是否在同一時刻出現模式切換
