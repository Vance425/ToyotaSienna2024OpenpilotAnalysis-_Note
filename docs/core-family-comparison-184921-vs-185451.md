# `0x260 / 0x115` 對照: `184921` vs `185451`

## 目的

在 `0x116` 和 `0x131` 已經縮出 phase/boundary 角色之後，這一輪確認：

- `0x260` 比較像什麼
- `0x115` 還值不值得排在前面

## `0x260` 比較

### `184921`

窗口內序列：

- `000042003e00...`
- `000042003d00...`
- `000044003d00...`
- `000045003d00...`
- `000047003d00...`
- `000049003d00...`
- `00004c003d01...`
- `00004f003e01...`
- `000053003e01...`
- `000055003e02...`
- `000057003e01...`
- `000059003f02...`
- `00005b003f02...`
- `00005c0040...`
- `00005c0041...`
- `00005b0042...`

觀察：

- 前 2 bytes 呈現平滑上升：`42 -> 44 -> 45 -> 47 -> 49 -> 4c -> 4f -> 53 -> 55 -> 57 -> 59 -> 5b -> 5c`
- 第 3-4 bytes 與 `0x131` 的後半狀態軸很接近：`003d -> 003e -> 003f -> 0040 -> 0041 -> 0042`
- 後 2 bytes 仍然像高變動尾端

判讀：

這不像單純保護 tag，而像「狀態主幹 + 變動尾端」。

### `185451`

窗口內序列：

- `00fef2f9...`
- `00fefbf9...`
- `00fefcf9...`
- `00fefdf9...`
- `00fefef9...`
- `00ff01f9...`
- `00feeff9...`
- `00ff04f9...`
- `00fef6f9...`
- `00fee6f9...`

觀察：

- 第 1 byte 幾乎固定 `00`
- 第 2-3 bytes 形成一個漂移中的中段狀態區：`fef2 / fefb / fefc / fefd / fefe / ff01`
- 第 4 byte 長時間固定為 `f9`，和 `0x131` 的 `f9xx` 背景軸同族
- 整體不像單一 counter，比較像 family state snapshot

判讀：

`185451` 裡的 `0x260` 雖然沒有 `184921` 那麼線性，但仍然像核心 family，同步反映事件帶切換。

### `0x260` 合併結論

`0x260` 目前最像：

- 一條核心 state/control family
- 不是單純 auth frame
- 也不像只是在旁邊跟著抖動

比較合理的角色是：

- `0x131` 提供邊界感
- `0x116` 提供 phase 感
- `0x260` 提供 family-level sync

## `0x115` 比較

### `184921`

窗口內例子：

- `07360002...`
- `073d0002...`
- `07430002...`
- `074c0002...`
- `074d0002...`
- `07550002...`
- `07570002...`
- `07580002...`
- `07530002...`
- `07500002...`

觀察：

- 第 1-2 bytes 會變
- 第 3-4 bytes 幾乎固定 `0002`
- 後 4 bytes 高熵

### `185451`

窗口內例子：

- `00000002d184...`
- `000000021623...`
- `000000026817...`
- `00000002a3be...`
- `00000002e237...`

觀察：

- 前 4 bytes 幾乎固定為 `00000002`
- 真正的變化大多在後 4 bytes

### `0x115` 合併結論

`0x115` 仍然有候選價值，但目前比較像：

- 同家族的伴隨 frame
- 或某種 payload 固定、尾端高熵的陪跑線

它不像 `0x116` 那樣直接把 phase 暴露出來，也不像 `0x260` 那樣有 family-level 狀態感。

## 目前角色分工

現在可以把主線收斂成：

- `0x131`: boundary
- `0x116`: phase
- `0x260`: core family sync
- `0x115`: secondary companion
- `0x610`: anchor
- `0xD5`: speed guardrail

## 接下來要看什麼

下一輪最值得看的，不再是全新的 ID，而是：

1. `0x260` 和 `0x131` 的中段 bytes 是否存在固定耦合
2. `0x260` 是否總是在 `0x116` phase change 前後一起轉折
3. `0x115` 是否可以降權，不再佔主要分析時間

如果只選一件事先做，建議先做：

- `0x260` 與 `0x131` 的結構對齊
