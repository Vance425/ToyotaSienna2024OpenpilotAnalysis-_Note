# CAN Log 研究計畫

## 目標

透過 CAN log 分析，找出 `2024 Toyota Sienna` 可取得 `TSK` 的方法，讓 `comma 3X` 有機會進入可用狀態。

## 研究問題

這份研究要回答四個問題：

1. 哪些 CAN 訊息與 Toyota security 流程有關
2. `comma 3X` 自動抓取失敗時，車上實際發生了什麼
3. 哪些 frame 看起來帶有 `SecOC/auth/counter` 特徵
4. 是否能從被動 log 中建立可重現的 TSK 反推流程

## 最低必要資料

- 車輛基本身份資訊
- 製造年月
- VIN 首碼
- 市場版本
- 測試時使用的軟體 / fork
- 測試時使用的線束型號
- 一組以上原始 CAN log

## 建議收集的 log 場景

### 場景 A: 原廠純上電

目的：

- 建立 baseline
- 觀察車輛在沒有 `comma 3X` 介入時的安全相關流量

### 場景 B: 接上 comma 3X 但未成功使用

目的：

- 找出自動抓取失敗時新增了哪些訊息
- 找出失敗前後的時序差異

### 場景 C: 重複多次開關機

目的：

- 找出每次啟動都會出現的固定模式
- 區分一次性噪音和真正關鍵序列

### 場景 D: 不同車輛狀態

例如：

- IGN on
- Ready
- 原廠 ACC on/off
- 原廠 ADAS 狀態切換

目的：

- 找出 security 流程是否與車輛狀態切換關聯

## 分析方向

### 1. 時序分析

先看：

- `comma 3X` 接入前後
- 啟動前後
- 錯誤發生前後

重點不是先猜 ID，而是先找「固定會出現的模式」。

### 2. 訊息分群

把可能相關的訊息分成：

- 啟動後短時間爆量出現的訊息
- 只在 `comma 3X` 連上時出現的訊息
- 只在失敗瞬間出現的訊息
- 疑似 request/response 成對訊息

### 3. 安全流程特徵

優先注意：

- 固定長度但內容每次不同的 payload
- 有 nonce / counter 感的欄位
- 同一個 core 下 tail 持續變化
- 特定 ECU 在固定狀態切換時才出現的封包

### 4. 差異分析

最重要的是比對：

- 原廠純車況 vs 接入 `comma 3X`
- 成功初始化前段 vs 失敗時段
- 多次測試之間的共同模式

## 預期產出

研究不一定立刻得到 TSK 本身，但至少要產出：

- 疑似安全相關 CAN ID 清單
- 可疑時序窗口
- 疑似 `SecOC/auth/counter` frame
- 自動抓取失敗的具體位置
- 下一輪驗證假設

## 成功定義

### 最小成功

知道 `comma 3X` 失敗卡在哪個 CAN 流程階段。

### 中等成功

縮小到少數幾個疑似與 TSK 有關的訊息群。

### 高成功

找到可重現的 TSK 取得方法或足以支持實作的證據鏈。
