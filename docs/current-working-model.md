# Current Working Model

## 目前最合理的角色模型

根據 `184921` 與 `185451` 兩段窗口的對照，現在最實用的工作模型是：

- `0x131`: 狀態邊界指標
- `0x116`: phase frame
- `0x260`: 核心 state/control family
- `0x115`: 次級伴隨候選
- `0x610`: 事件 anchor
- `0xD5`: 速度參考線

## 這個模型的意思

現在不要再把所有高分 ID 都當成同一種東西。

比較合理的是：

1. 先用 `0x610` 找事件點
2. 用 `0xD5` 排除速度事件
3. 用 `0x131` 找 reset/boundary
4. 用 `0x116` 看 phase 是否切換
5. 用 `0x260` 確認整個 family 是否同步改變

## 目前最值得相信的假設

### 假設 1

`0x116` 是最乾淨的 protected-phase 候選。

### 假設 2

`0x131` 不是單純數值訊號，而是很強的狀態邊界指標。

### 假設 3

`0x260` 不像純 auth tag，比較像核心 family 狀態框架。

## 下一步主線

所以接下來的分析主線應該是：

- `0x131 <-> 0x260` 的狀態耦合
- `0x116` 是否在這個耦合邊界上穩定切 phase

## 現在不需要優先做的事

目前不需要再花太多時間在：

- 擴大候選 ID 清單
- 回到傳統 UDS seed/key 流程
- 把 `0x115` 當第一主角
